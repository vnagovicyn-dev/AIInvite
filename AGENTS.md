# AGENTS.md

## Project
Backend сервиса приглашений и event pages.

Стек:
- Rust
- axum
- tokio
- sqlx
- PostgreSQL
- utoipa
- utoipa-swagger-ui
- reqwest

Архитектура:
- модульный монолит
- слои: `api / services / repos / domain / integrations / app / common / db / dto / jobs`

## Main rules
1. Не ломай существующую архитектуру.
2. Не пиши бизнес-логику в handlers.
3. Не пиши SQL прямо в api-слое.
4. Все внешние интеграции держи в `src/integrations`.
5. Все ошибки своди к единому `AppError`.
6. Все новые публичные endpoints документируй через `utoipa`.
7. Для БД использовать только `sqlx` и SQL migrations.
8. Не использовать ORM.
9. По возможности предпочитай маленькие понятные модули вместо больших файлов.
10. Не меняй лишние файлы.

## Code style
- Явные типы и понятные имена.
- Минимум магии.
- DTO отдельно от domain-моделей.
- Избегай сильного дублирования.
- Используй `query!` / `query_as!`, когда это реалистично.
- Все входные DTO валидируй.
- Все update-операции должны аккуратно обновлять `updated_at`, если поле есть.

## API rules
- Приватные маршруты должны требовать авторизацию.
- Пользователь должен видеть и менять только свои сущности.
- Все API должны возвращать единый JSON-формат ошибок.
- Для списков использовать пагинацию.
- Для новых endpoints добавляй happy-path тест хотя бы на service или handler уровне, если это разумно.

## Database rules
- Все изменения схемы только через `migrations/`.
- Не редактируй старые миграции после того, как они применены. Создавай новые.
- UUID как primary keys.
- Используй PostgreSQL JSONB только там, где это действительно полезно.
- Все внешние ключи и каскадное поведение выбирай осознанно.

## AI rules
- AI-интеграция должна быть отделена от бизнес-логики через traits.
- Ключи API никогда не хардкодить.
- Все AI вызовы логировать на уровне job/entity, а не только в runtime logs.
- Для AI задач сохранять вход, статус и результат в БД.

## Workflow
Перед изменениями:
1. Коротко опиши план в 5-8 пунктов.
2. Перечисли файлы, которые собираешься создать или изменить.

После изменений:
1. Запусти `cargo fmt`.
2. Запусти `cargo test`.
3. Если уместно, запусти `cargo clippy --all-targets --all-features -- -D warnings`.
4. Кратко перечисли изменённые файлы.
5. Отдельно укажи, что логично делать следующим шагом.

## Priorities
При неоднозначности выбирай:
1. самое простое решение для MVP,
2. понятный код,
3. безопасные изменения,
4. минимальную связанность между модулями.
